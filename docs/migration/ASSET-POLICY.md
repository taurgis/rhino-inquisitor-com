# Asset Policy

## Purpose

This document defines the Phase 3 asset-handling baseline for the Hugo scaffold. It sets the minimum image, CSS, JavaScript, and layout-stability rules that must hold before Phase 4 content migration begins.

## Scope

- Hugo-generated scaffold pages built into `public/`
- Page-resource and static-image usage in Phase 3 templates
- Local lab performance checks run with Lighthouse CI

Out of scope:

- Full WordPress media import
- CDN or edge image delivery
- AVIF-by-default processing
- Field Core Web Vitals monitoring after launch

## Image strategy

### Default path

- Prefer page resources for article and page-level images.
- Render scaffold template images through `src/layouts/partials/media/image.html`.
- For processable local raster images, the partial converts output to `webp` and emits explicit `width` and `height` attributes from the Hugo resource.
- For local non-processable resources that still expose metadata, the partial uses the original asset path and preserves explicit dimensions.

### Fallback behavior

- If an image does not resolve to a local Hugo image resource, callers must provide explicit `width` and `height` values.
- Images that cannot provide dimensions are not accepted in the Phase 3 scaffold path because they create avoidable CLS risk.
- Remote or third-party images are not part of the Phase 3 scaffold baseline.

### Format policy

- `webp` is the default processed format for local raster images in this baseline.
- AVIF remains a documented future enhancement only. The current repo baseline does not depend on undocumented AVIF processing behavior.

### Layout-shift prevention

- Every rendered `<img>` must include explicit `width` and `height` attributes.
- Images must remain responsive in layout via CSS such as `max-width: 100%` and `height: auto`.
- If a future embed or media block cannot provide explicit dimensions, reserve space up front with CSS `aspect-ratio` or another deterministic placeholder strategy before it is shipped.

## Asset budgets

These are uncompressed per-page budgets for the representative scaffold template set.

| Asset class | Budget | Baseline rule |
|---|---:|---|
| JavaScript | 50 KB | No hydration framework or non-essential runtime by default |
| CSS | 30 KB | Keep scaffold styling lean; avoid framework CSS payloads |
| Images | 500 KB aggregate | Representative pages should stay comfortably under this threshold |

## JavaScript policy

- No client-side framework hydration is allowed by default in the scaffold.
- Any new script must have a clear owner, justification, and review note before merge.
- Uncontrolled third-party scripts are not allowed in scaffold output.
- Inline JSON-LD generated by repo-owned SEO partials is treated as first-party metadata, not third-party runtime JavaScript.

## Performance gate model

### Field targets

- Largest Contentful Paint: `<= 2.5s`
- Interaction to Next Paint: `<= 200ms`
- Cumulative Layout Shift: `<= 0.1`

These remain field targets. They describe user-facing expectations and are evaluated from field data where available.

### Lab gate used in Phase 3

Phase 3 uses Lighthouse CI against the built static output in `public/`.

- LCP assert: `<= 2500ms`
- CLS assert: `<= 0.1`
- TBT assert: `<= 200ms`

TBT is used as the lab proxy for INP because Lighthouse does not measure real INP in lab runs.

## Representative template routes

The baseline performance gate runs on:

- `/`
- `/phase-3-performance-baseline/`
- `/category/platform/`

These routes are generated from scaffold-owned fixture content so the gate stays deterministic.

## Validation commands

- `npm run build:prod`
- `npm run check:perf:budget`
- `npm run check:perf`

## Verification expectations

- All generated `<img>` tags in scaffold output must carry explicit width and height.
- Representative pages must stay within the documented budgets.
- The generated HTML must not include uncontrolled third-party `<script src>` references.
- LHCI reports are written to `tmp/lhci/` for local review.
