## Active Risks

| ID | Description | Impact | Likelihood | Owner | Mitigation | Trigger | Contingency | Status |
|----|-------------|--------|------------|-------|------------|---------|-------------|--------|
| R-001 | Host duplication risk: apex and `www` variants are both directly reachable and indexable, which can split canonical signals and delay stable HTTPS behavior. | High | High | Migration Owner | Lock canonical host to `https://www.rhino-inquisitor.com/`, configure both apex and `www` correctly in GitHub Pages, point `www` directly to `<owner>.github.io`, and verify one-hop host canonicalization before launch. | Any release-candidate probe shows apex or non-canonical host returning indexable `200` content or a canonical URL outside `https://www.rhino-inquisitor.com/`. | Hold cutover, correct DNS and Pages custom-domain configuration, and re-run host/canonical validation before continuing. | Open |
| R-002 | Slash duplication risk: slash and no-slash variants can both return `200`, leaving duplicate crawl states if redirect normalization is not finalized. | High | High | SEO Owner | Preserve trailing-slash canonicals in generated output, validate canonical tags on all sampled routes, and decide in Phase 2 whether edge redirects are required for strict request-time normalization. | A sampled release-candidate route serves both slash variants as indexable `200` responses without an approved redirect or accepted canonical-only policy. | Escalate to the Phase 2 routing decision, add redirect enforcement at the chosen layer, and block launch if unresolved on traffic-bearing routes. | Open |
| R-003 | Legacy endpoint handling gaps: feeds, search, author, API, XML-RPC, and other system endpoints require explicit retirement or redirect behavior to avoid accidental indexable leftovers. | Medium | Medium | Migration Engineer | Keep the endpoint inventory explicit in the manifest and redirect plan, validate non-content endpoints in smoke tests, and ensure retirement behavior is assigned by implementation layer. | Any system endpoint, feed, or search route is missing from the final redirect/retirement matrix or still returns unexpected indexable content on the release candidate. | Block launch for unmapped high-signal endpoints, apply explicit redirect or `404` handling, and rerun endpoint verification before release. | Open |
| R-004 | Video and taxonomy parity risk: category, video, tag, and archive structures can diverge from Hugo output and break preserved URLs or canonical expectations. | High | Medium | SEO Owner | Validate parity for all approved `keep` category and video routes, carry taxonomy and archive sampling into later QA, and escalate any path mismatch before template freeze. | Any approved `keep` taxonomy or video URL lacks a matching Hugo route, returns the wrong canonical, or requires an unplanned merge decision. | Reclassify the affected route for explicit redirect handling, add targeted parity checks, and block launch if high-signal sections remain unresolved. | Open |
| R-005 | Redirect design may exceed GitHub Pages native capabilities because the documented Pages model covers static publishing and host-level domain redirects, not a general origin rewrite engine for arbitrary legacy path handling. | High | Medium | Migration Owner | Use `implementation_layer` as the normative planning field, separate Pages-static cases from edge/CDN cases early in Phase 2, and keep Google-preferred permanent redirects as the target where technically feasible. | Manifest or launch-plan requirements depend on path, case, slash, or header behavior that is not covered by the chosen hosting layer. | Add an external redirect-capable layer before launch or explicitly narrow redirect scope with SEO-owner approval and documented trade-offs. | Open |
| R-006 | DNS, certificate, and verification timing risk: GitHub Pages custom-domain reachability, HTTPS availability, and Search Console verification continuity must be sequenced correctly during cutover. | High | Medium | Migration Owner | Lower TTL ahead of cutover, validate DNS and CAA records, preserve Search Console verification methods, and treat delayed HTTPS availability as a launch hold condition rather than assuming immediate readiness. | Custom domain does not resolve correctly, `Enforce HTTPS` is unavailable inside the launch window, or verification continuity is lost after cutover. | Pause launch, correct DNS/certificate prerequisites, restart Pages certificate provisioning steps if required, and keep the legacy stack rollback-ready until stable. | Open |
| R-007 | Discovery coverage risk: Search Console enrichment added 86 URLs that were not present in inventory before enrichment, so any downstream workflow that relies on sitemaps or crawl output alone can silently miss live URLs. | High | Medium | Migration Engineer | Treat sitemap, crawl, and Search Console inputs as mandatory sources, and validate downstream Phase 2+ workflows against the normalized inventory rather than against sitemap-only exports. | A downstream script, review sample, or migration checklist excludes Search Console-only URLs or reintroduces sitemap-only assumptions. | Append missing URLs back into the active working set, rerun classification/report generation, and block the next phase gate until coverage is restored. | Open |
| R-008 | Link-equity visibility gap: the available Search Console links export only marked 36 URLs with external links while 1163 URLs remain `null`, so `null` cannot be treated as “no links” in retire decisions. | High | Medium | SEO Owner | Treat `has_external_links: null` as unknown, not false; require conservative review for retire decisions on high-priority URLs when backlink certainty is missing; and use richer backlink evidence if needed. | A retire or merge decision relies on `has_external_links: null` as proof that a URL has no external equity. | Escalate the URL for manual SEO review, obtain better link evidence where possible, and default to a safer keep/merge decision until the signal gap is resolved. | Open |
| R-009 | High-signal non-content retire queue risk: RHI-004 approved 132 high-priority `retire` records across `attachment`, `pagination`, and `system` classes with organic traffic and external-link signals, so generalized retirement rules can still hide route-specific exceptions. | High | Medium | Migration Owner | Keep the owner-review queue and summary outputs as active reference material, require explicit implementation planning for these classes in redirect design, and spot-check the highest-signal examples during Phase 6 and Phase 7 validation. | Any release-candidate plan still treats approved high-signal retire queues as generic class rules without route-level implementation review. | Re-open the affected subset for targeted SEO review, convert selected routes to bespoke redirects or preserved endpoints, and do not launch until the exceptions are documented. | Open |
