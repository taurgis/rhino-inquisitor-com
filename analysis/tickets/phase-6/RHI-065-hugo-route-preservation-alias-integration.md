### Goal

Implement deterministic route preservation and redirect outputs in the Hugo build by enforcing explicit `url` front matter on all preserved content and generating `aliases` only for approved redirect mappings from the finalized `migration/url-map.csv`. The output of this workstream is a Hugo build where every `keep` URL renders as a first-class content page, every approved `redirect` URL resolves through the chosen redirect mechanism with no chain, and every `retire` URL is handled according to the retirement policy (not silently remapped).

This workstream is the engineering implementation counterpart to WS-A (inventory) and WS-B (intent review). It must not begin before both those workstreams and the architecture decision (RHI-062) are `Done`.

---

### Acceptance Criteria

- [ ] Every `keep` URL in `migration/url-map.csv` has an explicit `url` front matter field in its Hugo content file matching the legacy path exactly
- [ ] Every approved `redirect` URL in `migration/url-map.csv` is implemented using the redirect mechanism chosen in RHI-062:
  - [ ] **If Model A:** Hugo `aliases` entry in the destination content file's front matter for each `legacy_url`
  - [ ] **If Model B:** Edge redirect rule entry per legacy URL, plus Hugo `aliases` as fallback for any alias-eligible paths
  - [ ] **If hybrid:** Each URL class implemented per the hybrid scope boundary defined in RHI-062
- [ ] Alias pages generated by Hugo include `<meta http-equiv="refresh">` pointing to destination AND `<link rel="canonical">` pointing to destination (verified via `cheerio`-based check)
- [ ] No `retire` URL is silently redirected to any destination — retired URLs must either resolve to Hugo's `404` page or be explicitly excluded from redirect outputs
- [ ] No redirect chain exists in the mapping (verified by graph check):
  - [ ] No `legacy_url` in the alias list is itself a `url` field that points to another alias
  - [ ] No alias page destination is itself an alias page
- [ ] Sitemap output (`public/sitemap.xml`) contains only final canonical destination URLs:
  - [ ] No legacy source URLs (`legacy_url` values from redirect records) appear in the sitemap
  - [ ] All sitemap URLs are absolute with canonical host (`https://www.rhino-inquisitor.com/`)
- [ ] `npm run check:url-parity` passes against the finalized URL manifest on a production Hugo build
- [ ] `scripts/phase-6/validate-alias-pages.js` exits with code 0 on the built `public/` directory:
  - [ ] Checks: alias page present for every approved redirect record, canonical tag correct, no self-referencing canonicals
- [ ] Hugo production build (`hugo --minify --environment production`) completes with zero errors

### Notes

- The `aliases` front matter in Hugo is a list of legacy paths that the content file should also respond to. Hugo generates a standalone redirect HTML page at each alias path containing a meta-refresh to the canonical destination. These pages are **not** 301/308 redirects at the HTTP level; they are static HTML pages served by GitHub Pages.
- The 2026-03-13 parity investigation also established an artifact-discipline requirement for this workstream: validate `npm run check:url-parity` against a clean production build output, because preview-host or stale `public/` artifacts can create false failures unrelated to current source behavior.
- This workstream must not begin until WS-B (RHI-064) is Done. Adding aliases from an unreviewed mapping carries significant SEO risk — a bulk alias assignment from an unchecked mapping can create thousands of low-quality redirect signals in one commit.
- The chain detection script should build a directed graph from the alias→url relationships and check for cycles and paths longer than one hop. Any path longer than one hop is a redirect chain and must be collapsed to a direct source→final-destination mapping.
- If Model B is chosen (RHI-062), the edge redirect configuration must be committed alongside or before alias pages, so both layers are consistent. Misaligned layers (where edge redirects to A but Hugo alias redirects to B) create canonical confusion.
- Reference: `analysis/plan/details/phase-6.md` §Workstream C, §Critical Corrections Applied in This Phase Plan
